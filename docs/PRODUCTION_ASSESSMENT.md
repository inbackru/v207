# Оценка готовности InBack к продакшн

_Независимая техническая оценка. Дата: июнь 2026._

---

## Итоговая оценка: **6.5 / 10 — Условно готов к запуску с оговорками**

Проект функционален и имеет богатую функциональность, но ряд критичных для продакшна компонентов требует доработки перед открытием публичного трафика.

## ✅ Что работает хорошо

### Функциональность
- Каталог ЖК с фильтрацией, поиском, картой — полностью рабочий
- Расчёт кэшбека, оформление заявок — работает
- SEO: sitemap, canonical, Open Graph, JSON-LD, 200+ SEO-страниц по городам и районам
- Мультигород: Краснодар, Сочи, Курск, Белгород, Майкоп — добавлены
- PWA, мобильная адаптация — есть
- Авторизация: Google, VK, ВКонтакте, Telegram Login Widget — реализованы
- Telegram-уведомления о заявках — работают при наличии `TELEGRAM_BOT_TOKEN`
- Глоссарий, ипотечный калькулятор, страницы льготных программ — есть
- Антиспам: CSRF-защита (Flask-WTF) на всех формах

### Техническая база
- PostgreSQL (не SQLite) — готова к нагрузке
- Gunicorn с несколькими воркерами — production-ready сервер
- Кэширование маршрутов (5 мин) — снижает нагрузку на БД
- APScheduler для фоновых задач (мониторинг цен и т.д.) — есть
- Логирование через стандартный Python logging

## ❌ Критические проблемы (нужно исправить до запуска)

### 1. Email-уведомления о заявках не работали
**Статус: ИСПРАВЛЕНО** в этой сессии.
- Функция `send_email_notification` теперь реально отправляет через SendGrid или SMTP
- **Требуется настройка:** добавить `SENDGRID_API_KEY` или `EMAIL_USER`/`EMAIL_PASSWORD` в секреты

### 2. Отсутствует мониторинг и алертинг
- Нет системы оповещения при падении сервера (uptime monitoring)
- Нет отслеживания ошибок (Sentry, Rollbar)
- **Рекомендация:** подключить бесплатный [UptimeRobot](https://uptimerobot.com/) + Sentry

### 3. Нет бекапов базы данных
- Если БД упадёт — данные о заявках, пользователях, ЖК будут потеряны
- **Рекомендация:** настроить ежедневный `pg_dump` с отправкой на S3/Backblaze:
  ```bash
  pg_dump $DATABASE_URL | gzip > backup_$(date +%Y%m%d).sql.gz
  ```

### 4. Нет rate-limiting на формы заявок
- Злоумышленник может отправить тысячи заявок подряд
- **Рекомендация:** добавить Flask-Limiter (50 заявок/час с одного IP)

## ⚠️ Важные проблемы (желательно исправить в первый месяц)

### 5. Данные о ЖК требуют обогащения
- Адреса: у ~40% ЖК нет округа/микрорайона (нужен прокси для ЦИАН)
- Фотографии: не у всех ЖК есть изображения
- Цены: актуальны только если запущен скрапер или ручное обновление

### 6. Производительность при высоком трафике не протестирована
- Нет нагрузочного тестирования
- Сложные SQL-запросы с 10+ JOIN (особенно в smart_search)
- **Рекомендация:** добавить индексы, включить `EXPLAIN ANALYZE` на медленных запросах

### 7. Секреты в коде
- `TELEGRAM_CHAT_ID='730764738'` захардкожен в `routes/applications.py` (строка 284)
- **Рекомендация:** перенести в переменную окружения `ADMIN_TELEGRAM_CHAT_ID`

### 8. Отсутствует политика конфиденциальности и согласие на обработку данных
- Формы собирают ФИО и телефоны — по 152-ФЗ нужна политика + галочка согласия
- **Риск:** административный штраф до 500 000 ₽ от Роскомнадзора

### 9. Кэш маршрутов не инвалидируется при изменении данных
- Если добавить новый ЖК через админку — он появится на сайте только через 5 минут
- **Рекомендация:** принудительно инвалидировать `_complexes_route_cache` при изменениях через админку
